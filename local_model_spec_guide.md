# 로컬 모델(Qwen, Gemma 등)을 위한 Spec(작업 지시서) 작성 가이드

로컬 모델(파라미터 크기가 상대적으로 작은 Qwen, Gemma 등)은 추론 능력(Reasoning)과 복잡한 지시 사항을 따르는 능력이 GPT-4나 Claude 같은 대형 모델에 비해 부족합니다. 따라서 모델 스스로 문제를 파악하고 넓은 범위의 코드를 수정하게 두면 환각(Hallucination)이 발생하거나 루프에 빠질 확률이 매우 높습니다.

이러한 로컬 모델의 한계를 극복하고 `Local Micro Agent`의 FSM 루프를 온전히 완주시키기 위해서는 **모델의 추론에 의존하지 않는, 극도로 구체적이고 결정론적인(Deterministic) Spec 문서(`run_spec.json` 등)**가 필요합니다. 모델이 "생각"할 여지를 없애고, 오직 "기계적인 치환 및 작성"만 하도록 통제하는 것이 핵심입니다.

---

## 1. 태스크의 극단적인 세분화 (Micro-tasking)

로컬 모델은 한 번에 여러 파일을 수정하거나 여러 로직을 융합하는 데 취약합니다. 작업의 단위를 쪼갤 수 있는 한 가장 작게 쪼개야 합니다.

*   **Bad:** "결제 로직을 리팩토링하고 관련 테스트 코드를 모두 업데이트하세요."
*   **Good:** 
    *   **Task 1:** `src/payment.py`의 `validate_card` 함수 내부에 잔액 확인 로직을 추가한다.
    *   **Task 2:** `tests/test_payment.py`에 잔액 부족 시 발생하는 예외 테스트 코드를 추가한다.
*   **Spec 적용:** `run_spec.json`의 `task_graph` 내 각 노드(Task)가 다루는 `deliverables`(수정 가능 파일)와 `read_hints`(참고 파일)를 1~2개 이하로 철저히 제한하세요. 컨텍스트가 좁을수록 로컬 모델의 집중력이 높아집니다.

## 2. 모호함 제거: `target_symbols`와 `edit_scope`의 명문화

모델에게 코드를 찾아보라고 지시하지 말고, **정확히 어디를 수정해야 하는지 핀포인트로 지정**해주어야 합니다.

*   `target_symbols` / `target_regions`: 모델이 전체 코드에서 위치를 찾다 헤매지 않도록 `src/parser.py::parse_item`과 같이 정확한 심볼(클래스나 함수명)을 명시하세요.
*   `edit_scope`: 추상적인 목적 대신 구체적인 동작을 적습니다.
    *   *Bad:* "보안 예외 처리를 강화할 것"
    *   *Good:* "입력값 `x`가 None일 경우 `ValueError('Invalid input')`를 발생시키는 분기문을 최상단에 추가할 것"

## 3. 테스트 합성(Acceptance Synth) 의존도 낮추기

`Local Micro Agent`에는 모델이 스스로 테스트를 합성(`ACCEPT_SYNTH`)하는 기능이 있지만, 로컬 모델은 종종 테스트 코드 자체를 잘못 작성하여 무한 루프를 유발합니다.

*   **해결책:** `acceptance.kind`를 `"synthesized"` 대신 `"command"`나 `"metric"`으로 고정하세요.
*   사람이 미리 작성해 둔 외부 테스트 스크립트나 구체적인 `pytest` 명령어를 Spec에 하드코딩(`test_commands`)하여, 컨트롤러가 모델의 결과물을 냉정하고 결정론적으로 채점하게 만들어야 합니다.

## 4. 하이브리드 Spec 생성 (Human-in-the-Loop 또는 대형 모델 활용)

로컬 모델을 사용하여 `SPEC_SYNTH`(Spec 자체를 합성하는 단계)를 진행하면 품질이 떨어지는 설계도가 나올 수 있습니다. 쓰레기 설계도(Garbage in)가 들어가면 이후 구현도 반드시 실패(Garbage out)합니다.

1.  **대형 모델 1회 사용:** `run_spec.json`을 최초로 생성할 때만 API(예: Claude 3.5 Sonnet, GPT-4o)를 사용하거나, 해당 역할을 수행하는 Lane만 똑똑한 모델로 분리(`spec_synth` role override)하세요. 이후 반복적인 `CODE` / `TEST` 루프 노동은 로컬 모델이 저렴하게 수행하도록 맡깁니다.
2.  **수동 템플릿 제공:** 사용자가 뼈대가 되는 Spec을 직접 작성하고, 빈칸(구체적 구현)만 로컬 모델이 채우도록 유도합니다.

## 5. 방어적 계약(Design Contract)과 롤백 설계

로컬 모델은 실패할 때 복구 방안을 스스로 창의적으로 떠올리지 못합니다. 실패 시 모델이 맹목적으로 따를 수 있는 기계적인 롤백 지침을 Spec에 포함하세요.

*   `fallback_plan`: "이전 코드로 롤백하고, 해당 함수 전체를 `pass`로 덮어씌운 뒤 경고 로그만 남긴다." 와 같이 실패 시 시도할 수 있는 가장 단순하고 바보 같은(그러나 시스템을 깨뜨리지 않는) 방안을 명시해야 합니다.
*   `correctness_rationale`: 모델이 스스로 코딩하면서 자신이 짠 코드가 왜 안전한지 되짚어볼 수 있도록 앵커(Anchor) 역할을 하는 텍스트를 제공하세요.

## 6. 어휘와 프롬프트 제어

로컬 모델은 복잡한 영단어나 추상적인 지시어를 무시하는 경향이 있습니다.

*   명령문은 짧고 간결하게 작성하세요.
*   "~를 고려하세요(Consider)", "~하면 좋습니다(Preferably)" 같은 유연한 표현은 금물입니다.
*   "~해야만 합니다(MUST)", "~하지 마시오(DO NOT)"와 같은 강제성 있는 어휘를 사용하여 제약 조건(Constraints)을 프롬프트나 Spec에 굵게 박아넣어야 합니다.

---

### 결론
로컬 모델을 사용한 에이전트 루프의 핵심은 **모델의 '지능'을 믿는 것이 아니라, 모델을 단순히 '텍스트 치환 및 타자 치는 기계'로 취급하여 완벽하게 통제된 작업 지시서(Spec) 위에 올려놓는 것**입니다. 시스템(컨트롤러)의 엄격함과 Spec의 구체성이 로컬 모델의 부족한 추론 능력을 훌륭하게 메울 수 있습니다.
